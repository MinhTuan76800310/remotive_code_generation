Recipes — How to Use, How to Add, Example Use Cases

A recipe is a named, deterministic ECU behavior pattern. The bmgen compiler picks a recipe by the pattern: field in your YAML handler, validates the handler's shape against that recipe, and renders a fixed Jinja2 template to produce the handler's Python method. This document is the complete reference: the dispatch pipeline, every existing recipe with a real example, and the step-by-step recipe for adding a new one.

1. Where a recipe sits in the compiler pipeline

A recipe is dispatched in two places and never elsewhere. Knowing both is the key to understanding the system.

Dispatch 1 — _apply_value_exprs
Runs after IR validation. For every non-novel_logic handler it looks up the recipe by pattern and calls recipe.output_value_expr(handler), stamping the result onto every OutputSignalIR.value_expr. This is the behavioral core: the single place a recipe says "this is the value I write".
builder.py:318
Dispatch 2 — build_template_context
For each handler it calls recipe.validate(handler_ir) (raises on error) then recipe.build_context(handler_ir) to produce the Jinja2 variables for the handler template. WebsocketBridge is dispatched separately here for each WebsocketListenerIR — not through the handler loop.
context_builder.py:88
Guard — Invariant 9
The validator rejects any pattern not in the registry, unless novel_logic: true is set. The known set is read from the registry itself, so registering a recipe is enough — no validator edit needed.
validators.py:198
Why two dispatches? output_value_expr (Dispatch 1) answers "what value?" and is also used by the verifier's value-expression checks. validate + build_context (Dispatch 2) answer "is the shape legal, and what template vars?". Splitting them means a recipe's semantics live in one method, and its template plumbing in another — adding a recipe rarely touches build_context boilerplate.

2. The Recipe contract (anatomy)

Every recipe is a subclass of the abstract Recipe base. Five members are mandatory; output_value_expr has a sensible default.

class Recipe(ABC):
    @property
    def name(self) -> str: ...            # unique registry key, e.g. "DirectSignalMapping"
    @property
    def description(self) -> str: ...     # one-line human text (shown by `bmgen recipes`)
    @property
    def template_name(self) -> str: ...   # Jinja2 file, e.g. "handler_direct.py.j2"
    def validate(self, handler_ir) -> list[str]: ...   # return [] if OK, else error strings
    def build_context(self, handler_ir) -> RecipeContext: ...  # Jinja2 vars for the template
    def output_value_expr(self, handler_ir) -> str: ...        # OPTIONAL — the value written
    def required_fields(self) -> dict: ...                     # OPTIONAL — for `bmgen recipes`

What each method owns

validateshape rules: input/output counts, state presence, periodic presence, threshold set
output_value_exprthe Python expression written to every output signal — the behavior
build_contexttemplate variables (names, refs, tuples, ns var placeholders)
template_namewhich handler_*.py.j2 renders this handler

The 6 handler templates

handler_direct.py.j2read 1 signal → update_signals. Used by Direct + Threshold
handler_logic.py.j2read N signals → update_signals. Used by 4 logic gates
handler_toggle.py.j2read 1 → toggle bool state → update_signals
handler_blink.py.j2read 1 → start/stop ticker → blink loop
handler_websocket.py.j2background ws loop → update_signals
handler_weighted.py.j2multi-frame fan-in → latch bools → Σ w·bool → update_signals. Used by WeightedLogOdds

3. How to use a recipe (3 steps)

① Pick a pattern

Run bmgen recipes to list every registered recipe with its requirements. Choose the one whose behavior matches your ECU logic.

cd remotive-bm-compiler
uv run bmgen recipes

② Write the handler

Set pattern: to the recipe name and provide the fields that recipe requires (inputs, outputs, and recipe-specific keys like threshold or state).

handlers:
  - name: on_seat_occupancy
    pattern: ThresholdMapping
    threshold: 8
    operator: ">="
    true_when: below
    input:  { namespace: ..., signal: ... }
    output: { namespace: ..., signals: [...] }

③ Generate + verify

Generate the model; the recipe's validate() runs during generation and fails loudly if the shape is wrong. Then run the 3-layer verifier.

uv run bmgen generate seat_ecu.yaml --out gen/
uv run bmgen verify gen/ --spec seat_ecu.yaml
One handler = one CAN frame. Every per-handler recipe fires on exactly one frame_filter. All input signals in a handler must live in that same frame — that is why LogicAnd reads LogicInputs.A and LogicInputs.B from one LogicInputs frame, not two. To fuse signals from two different frames, use the latch-and-recompute pattern with novel_logic: true (see the escape hatch).

Example use case — seat-weight child detection

A child is light, so SeatOccupied (child present) is TRUE when weight is BELOW 8 kg. The author's rule "False if weight ≥ 8 else True" is expressed as operator: ">=" + true_when: below → 1 if not (weight >= 8) else 0, with no hand-filled stub.

handlers:
  - name: on_seat_occupancy
    pattern: ThresholdMapping
    threshold: 8
    operator: ">="          # MUST be quoted — see warning
    true_when: below
    input:
      namespace: SEAT-CpdCan0
      frame_filter: SeatWeightSensor
      signal: SeatWeightSensor.WeightKg
    output:
      namespace: SEAT-CpdCan0
      signals:
        - SeatInput.SeatOccupied
YAML quoting trap. operator: >= unquoted is invalid YAML — > is the folded-block-scalar indicator, so it raises yaml.YAMLError at parse time, before any recipe validation can give a friendly message. Always quote: operator: ">=". Omitting operator entirely is valid (defaults to >).

A logic ECU that must compute several boolean gates over a single LogicInputs frame carrying flags A and B. Each gate is a separate handler on the same frame; the compiler reads N signals from that one frame per handler.

WeightedLogOdds
handlerstateful (latched)multi-frameCAD

Compute a weighted sum of bool(inputi) across multiple CAN frames (possibly different namespaces), compare to a threshold, and write 0/1 to one or more output groups. Values are latched in instance state so the output reflects the last seen state of every input, not just the one in the current frame. This is the recipe behind the CAD (Child-Activity-Detection) decision in the live Child Presence Detection pipeline.

Requirements

Inputs
≥ 1 weighted_inputs, each with namespace, signal, weight, frame_filter
Outputs
≥ 1 output group (multi-namespace OK)
State
latched — one bool per input is held in self._<var>_latched; on reboot defaults to False
Periodic
none
weighted_threshold
required float — the threshold θ that Σ(wi · bool(vi)) must meet or exceed
Pattern shape
YAML pattern: is a list whose first entry names the recipe and carries weights: + threshold:

Value written

1 if (w₁·bool(v₁) + w₂·bool(v₂) + …) ≥ θ else 0, with each vi = the last-seen bool of that input signal.

Generated handler

async def Classify_Child(self, frame: Frame) -> None:
    if "SeatInput.SeatOccupied" in frame.signals:
        self._seat_occupied_latched = bool(frame.signals["SeatInput.SeatOccupied"])
    if "SeatInput.AdultWeightPresent" in frame.signals:
        self._adult_weight_present_latched = bool(frame.signals["SeatInput.AdultWeightPresent"])
    _weighted_sum = (1.0 * (1 if self._seat_occupied_latched else 0)) + (-1.0 * (1 if self._adult_weight_present_latched else 0))
    await self.seat_ocs_cpd_can_0.restbus.update_signals(
        ("OccupantClass.ChildPresent", 1 if _weighted_sum >= 1.0 else 0),
    )

Example use case — CAD-style child-presence decision

The SeatECU has a seat-occupied flag and an adult-weight-present flag on the same CAN bus. The CAD rule says: "child present iff occupied AND NOT adult". Modeling it as (+1 · occupied) + (−1 · adult) ≥ 1 gives the same outcome with no hand-written boolean glue: occupied & child → 1, occupied & adult → 0, empty → 0. Negative weights are a first-class feature, not a workaround.

handlers:
  - name: Classify_Child
    pattern:
      - name: WeightedLogOdds
        weights:
          SeatInput.SeatOccupied: 1.0
          SeatInput.AdultWeightPresent: -1.0
        threshold: 1.0
    input:
      - namespace: SEAT-CpdCan0
        signal: SeatInput.SeatOccupied
      - namespace: SEAT-CpdCan0
        signal: SeatInput.AdultWeightPresent
    output:
      - namespace: SEAT-OCS-CpdCan0
        signals: [OccupantClass.ChildPresent]
One method, N input frames. main.py.j2 registers one create_input_handler per unique (namespace, frame_filter) pair, all pointing to the same method. Each invocation latches whichever signal is in frame.signals, then recomputes the sum and writes all output groups. So a WeightedLogOdds handler can fan-in across frames that arrive independently — it does not require a single composite frame the way LogicAnd does.

async def on_turn_stalk(self, frame: Frame) -> None:
    turn_stalk_signal = frame.signals["TurnStalk.TurnSignal"]
    if turn_stalk_signal and not self._blink_enabled:
        self._blink_enabled = True
        self._start_blinking_blink_enabled()
    elif not turn_stalk_signal and self._blink_enabled:

        self._stop_blinking_blink_enabled()

        await asyncio.sleep(1.0)
        blink_state = not blink_state

            ("TurnLightControl.LeftTurnLightRequest", 1 if blink_state else 0),
            ("TurnLightControl.RightTurnLightRequest", 1 if blink_state else 0),

The turn stalk is a boolean enable. When it goes true, the handler starts the ticker; when false, it stops the ticker and zeros the lights. The blink_output block names which signals the ticker flips — here, both turn-light requests.

async def _ws_loop_camera_child_detection(self) -> None:
    ws_url = "ws://localhost:1122"
    signal_map = [
        ("ChildDetected", "CPDCameraData.ChildDetected"),
        ("CameraAvailability", "CPDCameraData.CameraAvailability"),

Example use case — external camera service → CAN

A driver-monitoring ECU whose camera detector is an outside service pushing JSON over a websocket. The ECU bridges that stream onto the vehicle CAN bus so other ECUs can consume it as ordinary CAN signals. A normal CAN-frame handler (on_selftest) coexists with the listener — the two are independent.

  - name: camera_child_detection
    url: ws://localhost:1122
    output_namespace: DMS-SeatCan0

Why model-level, not a handler? Every handler is triggered by an incoming CAN frame (create_input_handler + FrameFilter). A websocket has no CAN frame, so it cannot be a handler. The listener is launched in main() inside the BehavioralModel context before run_forever(), and cancelled in a finally. The recipe enforces this with an isinstance guard so it cannot be mis-dispatched through the per-handler loop.

The escape hatch — novel_logic: true

When a handler's behavior does not match any registered recipe, mark it novel_logic: true. The compiler then generates a stub (pass) with comments listing the intended inputs/outputs, and you hand-fill the method body. This is how the CentralHPC child-detection fusion is modeled today: two signals live in two different CAN frames, so no single-frame recipe can AND them — instead each handler latches one signal into state and recomputes the alert.

What the stub looks like

async def on_seat_update(self, frame: Frame) -> None:
    # novel_logic: implement manually
    # pattern: SeatLatch
    # input: SeatInput.SeatOccupied from CENTRAL-CpdCan0
    # output: ChildAlert.ChildAlertActive to CENTRAL-CpdCan0
    pass

Rules

  • Exempts Invariant 9 (unknown pattern) — any pattern: string is accepted.
  • Does NOT exempt the namespace invariants — input.namespace and output.namespace must still reference real namespaces, or you get Invariant 3/4 errors.
  • The pattern: name is free-form (e.g. SeatLatch, CameraLatch) — it is only a label in the stub comment.
  • The generated stub is a starting point; the running code is hand-filled afterward.

Evidence: stub renderer python_generator.py:147 · invariant guard validators.py:198 · live use vehicle_functions/child_detection/central_hpc.yaml

5. How to add a new recipe

Adding a recipe is a 2-file change. You do not touch the builder, validators, context_builder, or templates — the registry is the single source of truth and Invariant 9 reads the known-pattern set from it automatically.

Step 1 — write the recipe module

Create bmgen/recipes/<your_recipe>.py subclassing Recipe. The example below is a realistic new recipe: read two analog signals, output their average.

# bmgen/recipes/average_mapping.py
"""AverageMapping recipe — average two analog inputs, write the float result."""

from __future__ import annotations
from bmgen.ir.model import HandlerIR
from bmgen.recipes.base import Recipe, RecipeContext


class AverageMappingRecipe(Recipe):

    @property
    def name(self) -> str:
        return "AverageMapping"

    @property
    def description(self) -> str:
        return "Average two analog input signals and forward the float result"

    @property
    def template_name(self) -> str:
        return "handler_logic.py.j2"   # reuse the multi-input template

    def validate(self, handler_ir: HandlerIR) -> list[str]:
        errors = []
        if len(handler_ir.input_signals) != 2:
            errors.append(f"AverageMapping requires exactly 2 inputs, found {len(handler_ir.input_signals)}")
        if len(handler_ir.output_signals) < 1:
            errors.append("AverageMapping requires at least 1 output signal")
        if handler_ir.state is not None:
            errors.append("AverageMapping is stateless but state was declared")
        if handler_ir.periodic_task is not None:
            errors.append("AverageMapping is stateless but a periodic_task was declared")
        return errors

    def output_value_expr(self, handler_ir: HandlerIR) -> str:
        a, b = handler_ir.input_signals[0].python_var_name, handler_ir.input_signals[1].python_var_name
        return f"({a} + {b}) / 2.0"          # ← the behavioral core

    def build_context(self, handler_ir: HandlerIR) -> RecipeContext:
        input_tuples = [(s.name, s.python_var_name) for s in handler_ir.input_signals]
        output_tuples = [(s.name, s.value_expr) for s in handler_ir.output_signals]
        return RecipeContext(
            handler_name=handler_ir.name,
            pattern=self.name,
            template_name=self.template_name,
            context={
                "handler_name": handler_ir.name,
                "input_tuples": input_tuples,
                "output_tuples": output_tuples,
                "output_namespace_var": "",   # filled by context_builder
            },
        )

    def required_fields(self) -> dict:
        return {
            "name": self.name,
            "description": self.description,
            "template": self.template_name,
            "required_input_count": 2,
            "required_output_count": "≥1",
            "requires_state": False,
            "requires_periodic": False,
        }

If none of the 6 existing templates fit, add bmgen/compiler/templates/handler_<your>.py.j2. The template receives exactly the keys you put in build_context().context plus output_namespace_var (filled by the context_builder). Reusing handler_logic.py.j2 (multi-input read → update_signals) or handler_direct.py.j2 (single-input read → update_signals) covers most stateless recipes.

Common gotchas

  • One handler = one frame. All inputs must share the frame_filter. Cross-frame fusion → novel_logic.
  • Quote operator in ThresholdMapping: operator: ">=", never bare >=.
  • State needs reset_value for Toggle and Blink recipes — validate rejects it otherwise.
  • cleanup: true is mandatory on periodic tasks and websocket listeners.
  • Multi-input uses signals: (list); single-input uses signal: (scalar). Both accepted by the builder.
  • WebsocketBridge is model-level — declare under websocket_listeners:, not handlers:.
  • WeightedLogOdds values are latched. The output reflects the last-seen bool of every input, not the current frame — that is the point, and the reason the recipe is stateful even though no explicit state: block is declared.
  • WeightedLogOdds YAML shape is a list. The pattern: field is a list whose first element names the recipe and carries weights: + threshold:. Inputs use the list form (input: [{namespace, signal}, …]) not the single-frame scalar.
  • novel_logic still needs namespaces — Invariant 3/4 are not exempted.
