# Spec ‚Äî Align `test_env/VF_child-detection` with `inc_schema/*.yaml` (WeightedLogOdds fan-in)

**Date:** 2026-07-03
**Status:** Approved (user, sections 1‚Äì5)
**Branch:** `service_oriented`

---

## 1. Problem

The `remotive-bm-compiler` recently added support for the new SWC-composition
schema under `vehicle_functions/child_detection/inc_schema/`, including
frame-inference, multi-namespace fan-in (the `WeightedLogOdds` recipe), and
the `ecu:`/`software_components:` composition model. The compiler changes are
merged and tested (138/138 tests passing on 2026-07-03) and the in-tree
generated artifacts under `vehicle_functions/child_detection/generated/`
reflect the new behavior.

`test_env/VF_child-detection/` is the integration test environment ‚Äî it ships
a docker-compose topology, a per-ECU behavioral-model Python package, a
camera websocket service, and a pytest integration suite. The model Python in
`test_env/.../models/*/python/*/__main__.py` is still the **legacy** output
(no WeightedLogOdds, no `ecu:` schema, no `IsMoving` field on the camera
bridge). The integration test in
`test_env/VF_child-detection/tests/test_child_detection.py` was written for
the legacy `seat_occupied AND camera_detected` fusion in CENTRAL and is
incompatible with the new `Œ£ w·µ¢ √ó bool(input·µ¢) ‚â• threshold` decision rule.

This drift means:

- A fresh `remotive topology generate` against the (unchanged) instance YAMLs
  in `test_env/` would still produce a working compose for the legacy models,
  but the topology would no longer match the spec the rest of the project is
  evolving toward.
- Anyone running `cd test_env && docker compose --profile tester up` and
  comparing to `vehicle_functions/child_detection/inc_schema/` would see two
  different (and contradictory) truth sources.
- The compiler's new WeightedLogOdds path is not exercised end-to-end against
  the real test environment.

**Goal:** bring `test_env/VF_child-detection` in line with the new
`inc_schema/*.yaml` spec, end-to-end: re-generate the model Python from the
new specs, extend the camera service with an `IsMoving` field, update the
integration tests, and re-emit the build output. After this change, the
integration tests should exercise the WeightedLogOdds fan-in path against
the live docker-compose topology and pass.

## 3. The plan ‚Äî what changes

### 3.1 Source specs (`vehicle_functions/child_detection/inc_schema/`)

**`SWC_CAD_logic.yaml`** ‚Äî re-author to the new weighting scheme decided
with the user:

```yaml
# CAD_logic = Œ£ (w·µ¢ √ó bool(input·µ¢)) ‚â• threshold
#   SeatOccupied  = 0.3  (light on seat, low-confidence)
#   ChildDetected = 0.7  (camera says it's a child, high-confidence)
#   IsMoving      = 0.3  (amplifier: child is squirming / active)
#   threshold = 1.0
# Effective truth table (sum ‚â• 1.0 ‚áí Alert ON):
#   seat cam mov | sum  | alert
#   0   0   0    | 0.0  | OFF
#   1   0   0    | 0.3  | OFF   ‚Üê light seat alone is not enough
#   0   1   0    | 0.7  | OFF
#   0   0   1    | 0.3  | OFF
#   1   1   0    | 1.0  | ON
#   1   0   1    | 0.6  | OFF
#   0   1   1    | 1.0  | ON    ‚Üê camera + motion (no seat) is enough
#   1   1   1    | 1.3  | ON
handlers:
  - name: CAD_logic
    pattern:
      - name: WeightedLogOdds
        weights:
          SeatInput.SeatOccupied: 0.3
          CameraInput.ChildDetectedByCamera: 0.7
          CameraInput.IsMoving: 0.3
        threshold: 1.0
    input:
      - namespace: CENTRAL-CpdCan0
        signal: SeatInput.SeatOccupied
      - namespace: DMS-CpdCan0
        signal: CameraInput.ChildDetectedByCamera
      - namespace: DMS-CpdCan0
        signal: CameraInput.IsMoving
    output:
      - namespace: CENTRAL-CpdCan0
        signals: [ChildAlert.ChildAlertActive]
```

Note: `AirbagStatusReport.AirbagStatus` is **removed** from `input:` and
`weights:`. The `AIRBAG-CpdCan0` namespace can stay in
`centralHPC.yaml`'s `namespace_types:` (it was added there in the previous
task as a cross-ECU fan-in composition point ‚Äî leaving it costs nothing and
keeps the file uniform with the other ECUs' namespace lists). We do not
remove it.

**`SWC_camera_data_preprocessing.yaml`** ‚Äî no structural change (the file
already declares the `IsMoving` ws_key). Add a comment block documenting
the two-field message contract.

### 3.2 Camera service (`test_env/VF_child-detection/camera_service/camera_ws_server.py`)

Extend the standalone WS server to broadcast **two** independently-toggleable
fields, `ChildDetected` and `IsMoving`:

- `_state` adds `"IsMoving": 0` (default).
- `build_message(detected, moving)` emits both fields in the JSON payload.
- WS message handler accepts inbound `{"IsMoving": 0|1}` (independent of
  `ChildDetected`) and updates `_state['IsMoving']` accordingly; same for
  `ChildDetected`.
- `_UI_HTML` adds a second button row "Moving / Still" (amber accent), a
  small status pill showing the current moving state, and the JS `send()`
  becomes per-field, sending `{...currentState, [field]: v}`.
- CLI adds `--moving 0|1` (default 0) for `fixed` mode reproducibility; in
  `auto` mode `IsMoving` stays at 0 (only `ChildDetected` toggles) so that
  test scenarios are deterministic unless `--ui` is used.

The host-binding behavior (`0.0.0.0:1122`) and the `WS_CAMERA_URL` env-var
override path used by the DMS instance YAML do not change.

For `centralhpc` specifically, the regenerated `__main__.py` will contain:

- 3 input-only `CanNamespace` fields with disambiguated names
  (`cpd_can_0` for CENTRAL, `dms_cpd_can_0` for DMS,
  `airbag_cpd_can_0` for AIRBAG ‚Äî see BehavioralModelIR.__post_init__ in

- 3 latches: `_seat_input_seat_occupied_latched`,
  `_camera_input_child_detected_by_camera_latched`,
  `_camera_input_is_moving_latched`.
- 3 `create_input_handler()` calls (one per (namespace, frame) pair), all
  dispatching to the single `on_CAD_logic` method.
- The `on_CAD_logic` method latches each input, computes
  `_weighted_sum = 0.7*latched1 + 0.3*latched2 + 0.3*latched3` and writes
  `1 if _weighted_sum >= 1.0 else 0` to `ChildAlert.ChildAlertActive`.

## 6. Risks and edge cases

- **Image rebuild cost:** the `model` docker stage is layer-cached per
  `MODEL_PATH`. Re-generating `__main__.py` invalidates only the COPY step,
  not the `uv sync` step that pulls `remotivelabs-topology`, so rebuild time
  should stay under a minute per ECU.
- **Camera service `auto` mode:** with `IsMoving` static at 0, the auto
  mode is reproducible for tests 2 and 4 (no motion ‚Üí no alert). For test
  1, 3, and 5 we inject `IsMoving=1` directly via the broker's restbus
  on the `DMS-CpdCan0` namespace (signal
  `CameraInput.IsMoving`) ‚Äî that way the pytest runs deterministically
  without the camera service needing to be running.
- **centralHPC namespace disambiguation:** the legacy test_env
  `__main__.py` uses `cpd_can_0` for the single CENTRAL namespace. The
  new `__main__.py` introduces `dms_cpd_can_0` and `airbag_cpd_can_0` as
  input-only fields. The BehavioralModel namespace list passed to
  `BehavioralModel(...)` includes all three; the existing
  `BehavioralModelIR.__post_init__` collision resolution in
  `remotive-bm-compiler/bmgen/ir/model.py:289-328` makes this safe.
- **No broker / Remotive CLI present in CI:** the spec does not require
  runtime verification to land; the AST + `bmgen parse` checks are
  sufficient evidence. A follow-up spec will cover the actual
  `docker compose --profile tester up` smoke test in CI.
