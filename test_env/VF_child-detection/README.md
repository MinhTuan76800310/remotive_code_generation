# VF Child-Detection — Remotive Topology

Integration-test topology for the **Child Presence Detection (CPD)** vehicle
function. Five ECUs cooperate on a single CAN bus (`CpdCan0`):

```
SEAT  ──SeatOccupied──┐
                       ├──► CENTRAL ──ChildAlert──► COCKPIT ──HmiChildWarning──► (HMI)
DMS   ──ChildDetected─┘        ▲                                   │
   (from ws://localhost:1122)  │                                   │
                               └──DriverDecision── COCKPIT ◄──HmiDriverAction
                               │
                               └──AirbagControlCommand──► AIRBAG ──AirbagActuatorState──► (actuator)
                                        ▲
                                        └── (AirbagStatusReport ◄── AIRBAG)
```

| ECU      | Python module          | Role |
|----------|------------------------|------|
| SEAT     | `seatecu`              | seat-weight threshold → `SeatOccupied` (weight < 8kg = child) |
| DMS      | `drivermonitoringecu`  | bridges external camera **websocket** → `ChildDetectedByCamera` |
| CENTRAL  | `centralhpc`           | fusion (LogicAnd) → `ChildAlert`; driver-decision → `AirbagControlCommand` |
| COCKPIT  | `cockpithmiecu`        | shows `HmiChildWarning`; captures driver `TurnAirbagOff` → `DriverDecision` |
| AIRBAG   | `airbagcontrolunit`    | reports `AirbagStatus`; applies `AirbagControlCommand` → `AirbagActuatorState` |

The behavioral-model Python was generated by `remotive-bm-compiler` from the
YAML specs in `vehicle_functions/child_detection/*.yaml` (see the "Regenerate"
section below).

---

## Prerequisites

- **Remotive Topology CLI**: `remotive --version` (this repo tested against `remotivelabs-cli 0.15.1`)
- **Docker** + Docker Compose
- **uv** (for the standalone camera websocket service and per-model lock files)

---

## 1. Generate the topology runtime

```bash
cd test_env/VF_child-detection

remotive topology generate \
  -f instances/main.instance.yaml \
  -f settings/can_over_udp.settings.instance.yaml \
  --name vf_child_detection \
  build
```

Inspect the resolved instance (optional, no build):

```bash
remotive topology show instance instances/main.instance.yaml
```

---

## 2. Run the topology

### 2a. With the integration tests (one-shot)

```bash
docker compose -f build/vf_child_detection/docker-compose.yml \
  --profile tester up --abort-on-container-exit --build
```

### 2b. Persistent (for manual monitoring / UI)

```bash
docker compose -f build/vf_child_detection/docker-compose.yml up --build
```

The Remotive web UI (signal dashboard) is served through the generated nginx
container — open the URL printed by `docker compose up` (look for the
`topology-broker` / nginx port mapping, typically `http://localhost:8080` or
similar; check `build/vf_child_detection/docker-compose.yml` for the exact
`ports:` mapping).

## 3. Real-time signal monitoring

The topology broker listens on `http://localhost:50051` (default). In a
separate terminal while the topology is running:

```bash
# List all namespaces
remotive broker signals namespaces --url http://localhost:50051

# Watch the full child-detection chain
remotive broker signals subscribe --url http://localhost:50051 \
  --signal SEAT-SeatCan0:ChildDetectionInput.SeatOccupied \
  --signal DMS-DmsCan0:ChildDetectionInput.ChildDetectedByCamera \
  --signal CENTRAL-AlertCan0:ChildAlert.ChildAlertActive \
  --signal COCKPIT-HmiCan0:HmiChildWarning.ChildAlertActive \
  --signal CENTRAL-CommandCan0:AirbagControlCommand.AirbagControlCommand \
  --signal AIRBAG-ActuatorCan0:AirbagActuatorState.AirbagActuated
```

## 5. Test scenarios (drive the chain manually)

```bash
# --- Scenario A: child detected (light seat + camera) ---
# Seat weight 4kg (< 8kg threshold) → SeatOccupied = 1
remotive broker restbus add --url http://localhost:50051 --frame SEAT-LocalSensorCan0:SeatWeightSensor --start
remotive broker restbus update --url http://localhost:50051 --signal SEAT-LocalSensorCan0:SeatWeightSensor.WeightKg:4.0

# Camera detects child (inject directly on DMS namespace)
remotive broker restbus add --url http://localhost:50051 --frame DMS-DmsCan0:ChildDetectionInput --start
remotive broker restbus update --url http://localhost:50051 --signal DMS-DmsCan0:ChildDetectionInput.ChildDetectedByCamera:1

# → expect CENTRAL ChildAlertActive=1, COCKPIT HmiChildWarning.ChildAlertActive=1

# --- Scenario B: no child (adult on seat) ---
remotive broker restbus update --url http://localhost:50051 --signal SEAT-LocalSensorCan0:SeatWeightSensor.WeightKg:75.0
# → SeatOccupied=0, AND-gate => ChildAlertActive=0

# --- Scenario C: driver turns airbag OFF ---
remotive broker restbus add --url http://localhost:50051 --frame COCKPIT-HmiCanInput0:HmiDriverAction --start
remotive broker restbus update --url http://localhost:50051 --signal COCKPIT-HmiCanInput0:HmiDriverAction.TurnAirbagOff:1
# → COCKPIT DriverDecision → CENTRAL AirbagControlCommand → AIRBAG AirbagActuatorState.AirbagActuated=1
```

## 6. The camera websocket service (ws://localhost:1122)

The DMS ECU (`drivermonitoringecu/__main__.py`) connects to an external camera
service at `ws://localhost:1122`, reads JSON messages, and publishes
`payload["ChildDetected"]` onto the CAN restbus as
`ChildDetectionInput.ChildDetectedByCamera`.

**Message contract** (the only field DMS reads is `ChildDetected`):

```json
{"ChildDetected": 1, "confidence": 0.95, "source": "passenger-camera-sim", "timestamp": 1719000000}
```

### Run it standalone (bare-metal — ECU also bare-metal)

```bash
cd test_env/VF_child-detection/camera_service
uv run camera_ws_server.py                 # auto-toggles 0/1 every 2s (demo)
uv run camera_ws_server.py --value 1       # constantly "child detected"
uv run camera_ws_server.py --value 0       # constantly "no child"
uv run camera_ws_server.py --interval 1.0  # faster
```

### Run it when the ECU is in Docker

The ECU container's `localhost` is itself, so point it at the host:

- The `driver_monitoring_ecu.instance.yaml` already injects
  `WS_CAMERA_URL=ws://host.docker.internal:1122` + `extra_hosts` mapping.
- Start the camera service on the **host** with `uv run camera_ws_server.py`
  (binds `0.0.0.0:1122`). The DMS container will reach it via
  `host.docker.internal`.

### Stop / reset restbus

```bash
remotive broker restbus stop   --url http://localhost:50051 --namespace SEAT-LocalSensorCan0
remotive broker restbus reset  --url http://localhost:50051 --namespace SEAT-LocalSensorCan0
```

## 8. Regenerate the behavioral models from YAML

The Python in `models/*/python/` was generated from
`vehicle_functions/child_detection/*.yaml` by `remotive-bm-compiler`:

```bash
cd remotive-bm-compiler
SPEC=../vehicle_functions/child_detection
for f in central_hpc seat_ecu driver_monitoring_ecu cockpit_hmi_ecu airbag_control_unit; do
  uv run bmgen generate "$SPEC/$f.yaml" --out "_generated_v2/$f"
done
```

Then copy each `models/<ecu>/python/<package>/` from the generated output.

> **Note on the DMS model**: the generator emits a hardcoded
> `ws://localhost:1122`. The checked-in copy reads `WS_CAMERA_URL` from the
> environment (default `ws://localhost:1122`) so the same image works
> bare-metal and in Docker. If you regenerate, re-apply that one-line change in
> `drivermonitoringecu/__main__.py`.
